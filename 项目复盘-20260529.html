二、踩过的坑

1 部署平台换了 4 次

平台结果根因
Render需绑卡免费层也要信用卡验证身份
Railway需付费$5/月起，无免费额度
Fly.io新号受限免费层也有机器配额，新号提示超额
Render（绑卡后）成功$1 临时冻结验证，几天后退回
教训：下次先花 5 分钟看 billing 页面，确认是否免费、是否绑卡，再动手注册。

2 Vercel 多项目检测卡住

仓库里同时有 frontend/ 和 backend/ 目录，Vercel 自动进入多服务模式，强制要求 vercel.json，不能用简单的单项目配置。

解决：创建 vercel.json + .vercelignore 忽略 backend 目录，把根目录构建命令指向 frontend。

3 环境变量第一次没生效

在 vercel.json 顶层配了 env，但 Vite 构建时读取的是 .env.production 文件（Vite 约定），第一次部署前端打不开。

解决：创建 frontend/.env.production，Vite 生产构建时自动加载。

4 部署成功页看不到地址

Vercel 的 Congratulations 页面不直接显示域名，需要进 Deployments 详情页才能找到。

5 没先本地验证就部署

部署上去打不开才回去排查，浪费了好几轮重部署。应该先 npm run build && npm run preview 在本地确认无误。

四、核心心得

部署的真正瓶颈：选平台

不是写配置、不是改代码，而是一开始没搞清楚"免费"的附加条件（绑卡、配额、付费墙）。下次先花 5 分钟确认 billing 页面，再动手注册。

配置比代码容易出错

后端代码写完就没动过一行，部署全程在调 vercel.json、.env.production、render.yaml 这些配置文件。环境变量是最容易翻车的地方。

SSH 优于 Token

GitHub 推送认证，SSH Key 一次性投入，永久有效。Token 会过期，会忘记存在哪，会权限不够。

先跑通最小路径

如果一开始就从 Render 手动 Web Service + Vercel 简单配置入手，而不是在 Blueprint、Railway、Fly.io 之间来回跳，能省一半时间。最短路径验证通了再优化。

一句话总结：一个学习项目的上线之路，最难的从来不是代码——是"别人的服务怎么用"。
